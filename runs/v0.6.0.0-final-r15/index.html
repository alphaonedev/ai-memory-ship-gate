AI NHI analysis · Claude Opus 4.7

PHASE 2 FULLY GREEN — federation fanout fix validated under real load

Federation bug resolved. First run ever with Phase 2 at 100% convergence on all three nodes. Phase 3 migration also PASS. Phase 4 exposed a separate chaos harness config bug.

What this campaign tested

Phases 1/2/3 as before, then Phase 4 for the first time (all earlier runs died before reaching it).

What it proved (or disproved)

The federation fanout fix from PR #309 is effective under real load: 200 writes × 3 nodes = 600 rows measured, 100% landed everywhere. The migration roundtrip (SQLite → Postgres → SQLite, 1000 memories, idempotent) works. Phase 4 surfaced that the real-infra port/host overrides in phase4_chaos.sh collapsed all three chaos processes onto the same port — harness bug, not a product regression.

For three audiences

Non-technical end users

The data-loss bug we found and fixed in the last round: confirmed fixed. All three servers received all 200 messages. No surprises. The remaining failure is in our test rig for the chaos tests, not in the product.

C-level decision makers

Phase 2 convergence: 200/200/200. The silent data-loss class of bug identified in run 14 is demonstrably closed. First run ever where Phases 1, 2, and 3 all PASS simultaneously — a release-eligible configuration is two harness fixes away.

Engineers & architects

Phase 2 `reasons: []`, counts `{a: 200, b: 200, c: 200}`, probe1 201, probe2 503 — textbook. Phase 4 failure traces to phase4_chaos.sh overriding N0_PORT/N1_PORT/N2_PORT all to 9077: run-chaos.sh is a LOCAL 3-process harness by design, so collapsing ports caused bind collision. Separately, N0_HOST/N1_HOST/N2_HOST env vars are never read by run-chaos.sh (hardcoded to 127.0.0.1). Aspirational-override pattern broke the test.

Bugs surfaced and where they were fixed

  1. Phase 4 harness collapsed all three ai-memory processes onto port 9077

    Impact: Only one of three chaos processes could bind; the rest silently failed. Every subsequent write was counted as fail (no quorum possible).

    Root cause: phase4_chaos.sh set all N*_PORT=9077 intending remote nodes, but run-chaos.sh is local-by-design.

    Fixed in:

    • ship-gate commit 7adb773 (drop overrides)

What changed going into the next campaign

r16 drops the overrides, letting run-chaos.sh spawn three local processes on default ports 19077/19078/19079 as designed.
