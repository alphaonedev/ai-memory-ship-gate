AI NHI analysis · Claude Opus 4.7

Metric returns 6.286 — impossibly high — and partition_minority crashes at cycle 13

Two more chaos harness bugs: per-cycle DB + namespace reuse (counts cumulative across cycles → ratios >1) and SIGTERM graceful-shutdown races with next cycle's spawn.

What this campaign tested

Phase 4 with the surviving-peer metric, same two fault classes.

What it proved (or disproved)

Disproved that run-chaos.sh's cycle isolation was clean. The reused namespace meant each cycle's count query returned prior cycles' memories too. SIGTERM + `wait` didn't guarantee the listen socket released before the next spawn.

For three audiences

Non-technical end users

We kept finding problems in the way we ran the chaos tests. None of them were product bugs, but they were blocking us from knowing whether the product passed the chaos tests. Each cycle needs to be a clean slate, or the counts bleed into each other.

C-level decision makers

Harness hardening continues to surface ahead of the product assertions — a good posture. Nothing yet has pointed at a real product regression beyond the run-14 federation fanout fix. The chaos harness needed per-cycle isolation + deterministic teardown; both now in place.

Engineers & architects

Two stacked harness bugs. (1) DB path was `WORKDIR/node-N.db` shared across cycles — abandoned WAL from one cycle's SIGKILL'd primary fought the next cycle's replay. Plus cumulative namespace made count_node1 per-cycle = 3 × cycle_number, so the new `min(...)/total_ok` metric summed to 6.286. (2) SIGTERM at teardown kicked ai-memory's 30s graceful-shutdown path (red-team #233); the listen socket stayed bound past `wait`'s return. Next cycle's spawn collided.

Bugs surfaced and where they were fixed

  1. Per-cycle DB + namespace reuse polluted counts + crashed recovery

    Impact: Metric output 6.286 (impossible > 1). partition_minority died at cycle 13 with node-0 failed to start.

    Root cause: run-chaos.sh reused `node-0.db`/`node-1.db`/`node-2.db` and `namespace="chaos"` across all cycles. Abandoned WAL recovery raced with spawn.

    Fixed in:

    • PR #312 (MERGED)

  2. SIGTERM teardown race with next cycle's spawn

    Impact: Under partition_minority, listen socket held by gracefully-shutting-down process collided with next cycle's spawn.

    Root cause: ai-memory's graceful-shutdown path holds the socket during a WAL checkpoint that can take seconds. `wait` returns before the socket actually releases.

    Fixed in:

    • PR #312 (MERGED) — SIGKILL + 100ms settle

What changed going into the next campaign

r19 uses per-cycle `c${n}-node-${idx}.db` files + `chaos-c${n}` namespaces + SIGKILL teardown + 100ms settle.
