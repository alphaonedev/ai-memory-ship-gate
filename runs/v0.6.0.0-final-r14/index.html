AI NHI analysis · Claude Opus 4.7

Convergence STILL missed with 90s settle — federation fanout abort identified

Product bug uncovered. The harness is now correct; the product itself is losing writes to one peer per write.

What this campaign tested

4×50-write burst, 90s settle, full probe sequence.

What it proved (or disproved)

Even with 3× the settle time, node-B and node-C stuck at 83% / 82%. Pattern matched exactly: each non-leader peer received ~100 of 200 writes. Diagnosis: under W=2/N=3, only ONE peer acks synchronously before the leader returns 201, and the OTHER peer's in-flight POST is canceled by `joins.shutdown().await` once quorum is met.

For three audiences

Non-technical end users

We found a real bug in the product itself, not just the test. Imagine sending a package to two warehouses for redundancy — the delivery truck stopped as soon as one warehouse signed for it, leaving the other still waiting. One package, one warehouse. If something happens to that warehouse before the second delivery catches up, you lose the package. The fix makes the truck finish all deliveries before driving away.

C-level decision makers

Silent data-loss risk identified and scoped. If a customer runs the 3-node federation configuration, approximately half of their writes would only have one redundant copy instead of two. A single node failure during a write burst could have lost data. This is exactly the class of bug the ship-gate exists to catch. Discovery cost: zero customer impact.

Engineers & architects

`src/federation.rs::broadcast_store_quorum` spawns fanout POSTs in a JoinSet, waits for W-1 acks in a loop, then calls `joins.shutdown().await` which ABORTS still-in-flight tasks. Under W=2/N=3 with two concurrent fanouts racing, the loser's reqwest POST was being killed mid-flight — often before the receiving axum handler could commit. Net effect: every write landed on leader + 1 peer, leaving the other peer ~50% behind.

Bugs surfaced and where they were fixed

  1. Federation post-quorum fanout aborted mid-POST (silent data-loss risk)

    Impact: Each write landed on leader + only ONE peer instead of all three. No sync-daemon in Phase 2 means permanent lag. A node failure between quorum-ack and catch-up would lose the write.

    Root cause: `tokio::task::JoinSet::shutdown().await` after early-exit canceled in-flight spawned tasks. Detaching them instead preserves delivery.

    Fixed in:

    • PR #309 (MERGED)
    • Ship-gate issue #1 (CLOSED)

What changed going into the next campaign

PR #309 lands on release/v0.6.0. r15 validates: convergence should be 200/200 on all three nodes.
